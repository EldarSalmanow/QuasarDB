# Руководство контрибьютора

Добро пожаловать в проект **QuasarDB**!
Чтобы наша работа была слаженной, код оставался чистым, а конфликты слияния сводились к минимуму, мы придерживаемся строгих правил разработки. Пожалуйста, внимательно прочитайте этот документ перед началом работы.

---

## 1. Стратегия ветвления

Мы не пушим напрямую в `main`! Вся работа ведется в изолированных ветках и вливается через Pull Requests (PR).

### Защищенные ветки:
*   **`main`** — Продакшен-ветка. Здесь лежит только стабильный, протестированный код, который можно сдавать преподавателю. Прямые коммиты сюда запрещены.
*   **`stage/zero`** — Текущая ветка интеграции для MVP (Задачи Уровня 0). Все новые фичи базового функционала вливаются сюда. Когда Уровень 0 будет готов, мы сделаем PR из `stage/zero` в `main`.

### Как называть свои ветки:
Формат: `<тип>/<номер-jira>-<краткое-описание>`
*   Разработка новой фичи: `feature/STORAGE-3-b-tree-index`
*   Исправление бага: `bugfix/PARSER-2-fix-create-table`
*   Настройка инфраструктуры: `chore/INFRA-4-devcontainer`

## 2. Правила коммитов

Мы используем стандарт[Conventional Commits](https://www.conventionalcommits.org/). Сообщения должны быть на английском языке и обязательно содержать номер Jira-задачи в квадратных скобках.

**Формат:** `type(scope): [JIRA-TICKET] message`

**Типы (type):**
*   `feat`: Новая функциональность (например, добавление оператора AND).
*   `fix`: Исправление ошибки.
*   `refactor`: Переписывание кода без изменения функционала.
*   `test`: Добавление или исправление тестов.
*   `chore`: Обновление сборки, CMake, зависимостей, форматирование.
*   `docs`: Изменение README или документации.

**Примеры идеальных коммитов:**
*   ✅ `feat(storage):[STORAGE-1] implement pager for binary disk IO`
*   ✅ `fix(parser): [PARSER-3] fix memory leak in AST tree destruction`
*   ✅ `chore(infra): [INFRA-1] add gRPC dependencies via Conan`
*   ❌ `fixed bug in parser` *(неправильно: нет типа, скоупа и номера задачи)*

## 3. Как взять задачу в работу

1. Перейдите в нашу **Jira** и переведите задачу в статус `In Progress`.
2. Обновите локальный репозиторий:
   `git checkout stage/zero`
   `git pull origin stage/zero`
3. Создайте ветку от `stage/zero`:
   `git checkout -b feature/JIRA-123-my-feature`
4. Напишите код. **Обязательно добавьте Unit-тесты для вашей логики.**
5. Сделайте коммиты по правилам выше.
6. Отправьте ветку на GitHub: `git push origin feature/JIRA-123-my-feature`
7. Откройте Pull Request (PR) на GitHub. В качестве базовой ветки (`base`) выберите `stage/zero`.

## 4. Pull Requests и Code Review

Каждый PR должен быть проверен минимум одним участником команды (Reviewer) и одобрен Team Lead'ом.

**Правила для автора PR:**
*   В описании PR укажите, что было сделано. Если PR закрывает задачу, напишите `Closes #НомерЗадачи`.
*   Убедитесь, что все проверки CI светятся зеленым (✅).
*   PR не должен быть огромным. Идеальный размер — до 500 строк измененного кода. Если задача огромная (как REVERT), разбивайте её на 2-3 PR'а.

**Правила для ревьюера:**
*   Проверьте читаемость кода и логику алгоритмов.
*   Если есть замечания — оставляйте комментарии к конкретным строкам кода.
*   Не придирайтесь к форматированию (пробелы, скобки) — это задача линтера в CI.

## 5. Что проверяет CI/CD (GitHub Actions)?

Когда вы создаете PR, GitHub автоматически запускает облачный сервер (пайплайн). Ваш код не может быть слит (merged), если CI упал ❌.

Пайплайн делает 4 вещи:
1.  **Build:** Компилирует ваш код (используя Ninja и g++-13). Проверяет отсутствие ошибок синтаксиса.
2.  **Clang-Format:** Проверяет, что ваш код отформатирован по стандарту Google. *Если CI упал здесь — просто нажмите Ctrl+S (Save) в VS Code с включенным автоформатированием и сделайте новый коммит.*
3.  **Clang-Tidy:** Запускает статический анализатор. Он упадет, если вы забыли инициализировать переменную, оставили утечку памяти или написали потенциально опасный код (`bugprone-*`). Исправьте замечания линтера!
4.  **Tests:** Запускает все E2E и Unit-тесты (GTest). Если вы сломали чужой код, тесты это покажут.

## 6. Требования к локальной среде разработки

Чтобы не воевать с ошибками компиляторов и версиями CMake:
1. Установите Docker Desktop и VS Code.
2. Откройте папку проекта в VS Code.
3. Нажмите `F1` -> введите **`Dev Containers: Reopen in Container`**.
4. Среда сама установит нужные линтеры, компилятор C++20 и gRPC. Вся команда разрабатывает проект в абсолютно идентичных условиях!
